# Agent 指令语料采集表

这个文档只需要你填写两类内容：

1. 你在飞书里会怎么说。
2. 你希望 Agent 怎么回复、弹什么卡片、执行什么结果。

其余内容由 Codex 负责：
- 判断现有系统是否已经可行。
- 分析需要调用哪些工具。
- 标出缺口和风险。
- 把还不可行的语料推进到可行。
- 补测试、改路由、改工具、改确认卡或澄清卡。

填写说明：
- “希望拿到的回复/效果”可以直接写口语描述。
- 如果是写操作，请写清楚你希望看到什么确认卡。
- 如果你不确定正确回复应该长什么样，只写“希望他先问我什么”也可以。
- “Codex 分析”和“推进状态”两列你不用填。

状态约定：
- `待分析`：你刚填，我还没处理。
- `已可行`：当前系统已经能做到。
- `需改造`：方向明确，需要开发。
- `已推进`：已经改完，待你飞书验收。
- `暂缓`：需要更多业务信息或依赖其它模块。

## 0. 复合任务 / 多步骤目标（重点）

复合任务不是边缘场景，而是后续 Agent 的主要使用方式。你可以优先填写这部分。

适用：一句话里包含多个目标，或者需要先查询、再判断、再执行，例如：
- 先找数据最好的链接，再按这个 ID 复制新链。
- 先查一组商品表现，再分别处理。
- 先看审计记录，再回滚某一次任务。
- 先定位同款组，再对整组做运营动作。

当前落地口径：
- Agent planner 优先接管未知自然语言指令，可返回 2-8 步多步骤计划。
- 前序步骤的结构化结果可以通过 `${rank.bestProductId}` / `${steps.rank.sameSkuGroupId}` 传给后续步骤。
- 普通写操作会先停在 Agent 确认卡；确认后执行当前步骤，并继续执行剩余步骤。
- 如果剩余步骤里再次遇到写操作，会再次停住要求确认，不会因为第一次确认而批量执行所有写操作。
- `rental.priceChange`、`rental.specRemovePlan`、`rental.newLinkBatchPlan` 属于预览/计划型工具，只会生成专用确认卡，不会在专用确认前改价、删规格或复制新链。
