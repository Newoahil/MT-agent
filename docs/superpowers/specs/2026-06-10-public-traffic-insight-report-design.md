# 公域数据日报洞察增强 Design

## Goal

将当前公域数据日报从“数据接通版”升级为“经营决策版”：飞书卡片能让老板快速看到昨日对比、核心问题和建议动作，Markdown/XLSX 能让运营复盘完整明细并直接分配执行。

## Problem

当前日报已能抓取并聚合公域曝光页和公域访问数据页，但输出信息密度不足：

- 飞书卡片只展示漏斗、模块数量和三个 Top5，缺少经营结论。
- 多个模块容易显示为 0 或“无”，没有解释为什么为空。
- 建议操作被混在 `action` 字段里，飞书只展示 `reason`，导致看不到可执行动作。
- 新聚合分析中 `newProductObservation` 和 `lifecycleGovernance` 目前为空，无法列出新品观察和生命周期治理商品。
- 1日指标被过度用于分类判断，7日和30日没有充分参与摘要和治理判断。

## Scope

包含：

- 新增“经营结论”数据结构，输出多条较昨日变化的叙述。
- 新增独立“建议操作”板块，飞书、Markdown、XLSX 都展示。
- 恢复并增强新品观察、生命周期治理模块。
- 问题模块保留原因说明，同时给出对应动作建议。
- 模块为空时输出解释性文案，而不是只写“无”。
- 保留现有抓取流程、中文文件命名和 `output/YYYY-MM-DD/` 存储方式。

不包含：

- 自动修改商品。
- 飞书按钮交互。
- LLM 自动生成长文案或问答。
- 新增抓取页面。
- 改变支付宝登录方式。

## Design

### Report Context

扩展 `PublicTrafficDataReportContext`：

- `conclusions`: 今日经营结论列表。
- `recommendedActions`: 独立建议操作列表。
- `emptySectionNotes`: 各模块为空时的解释文案。

建议操作继续复用 `PublicTrafficReportSectionItem` 的基本形态，但语义明确为：

- `identifier`: 商品或模块标识。
- `action`: 要做什么。
- `reason`: 为什么要做。

### Yesterday Comparison

结论部分使用“今日 1日 summary”和“昨日 1日 summary”做对比。昨日 summary 优先从 `output/<yesterday>/公域数据上下文_<yesterday>.json` 读取。

如果昨日上下文存在，输出多条较昨日叙述：

- 曝光较昨日上升/下降/持平，展示差值和百分比。
- 公域访问较昨日上升/下降/持平，展示差值和百分比。
- 金额较昨日上升/下降/持平，展示差值和百分比。
- 发货较昨日上升/下降/持平，展示差值和百分比。
- 曝光到访问率、访问到发货率较昨日变化，展示百分点变化。

如果昨日上下文缺失，结论仍输出今日概况，并注明“暂无昨日公域数据上下文，今日仅展示基准值”。这不能阻断日报生成。

### Analysis Rules

问题和机会模块使用不同周期分工：

- 1日：发现今日需要立即关注的问题。
- 7日：判断趋势是否稳定，避免单日波动导致误判。
- 30日：判断长期弱表现和生命周期治理。

模块策略：

- 曝光不足：1日曝光低且7日曝光弱，或已托管但近期几乎无访问。建议检查托管状态、标题、主图、类目和是否继续投放。
- 曝光有但点击弱：1日或7日曝光达到阈值但公域访问率低。建议优化主图、标题、价格露出和首屏卖点。
- 点击有但转化弱：访问达到阈值但订单或发货弱。建议检查租赁价格、押金、库存、风控、履约链路。
- 高潜力商品：曝光、访问、发货或金额表现相对好。建议继续放量，并复制标题/图片/价格结构到同类商品。
- 新品观察：来自日差分 `new_product` 或近期端内 ID 映射新增商品。建议观察 3-7 天，重点看曝光、访问和首单/发货。
- 生命周期治理：托管天数达到阈值且30日曝光、访问、金额弱。建议下架、替换或重做素材。

### Recommended Actions

新增聚合后的独立建议操作板块。来源包括各模块 Top 商品，按优先级排序：

1. 点击有但转化弱且访问量高。
2. 曝光有但点击弱且曝光量高。
3. 生命周期治理。
4. 高潜力商品放量。
5. 新品观察。
6. 曝光不足。

每条建议操作格式要能直接执行：

- `端内ID 565：检查价格/押金/库存/履约链路。原因：1日后链路访问 132，发货 0。`
- `端内ID 661：优化主图和标题首屏卖点。原因：7日曝光高但访问率低。`

飞书卡片最多展示 8 条建议操作；Markdown/XLSX 展示完整 TopN。

### Output Changes

飞书卡片结构调整为：

1. 今日经营结论。
2. 今日漏斗。
3. 建议操作。
4. 问题与机会模块数量。
5. 模块 Top 列表：曝光不足、点击弱、转化弱、高潜力、新品观察、生命周期治理。
6. 报告文件。

Markdown 结构调整为：

- 经营结论。
- 1日/7日/30日总览。
- 建议操作。
- 六个模块明细。
- 商品明细仍留在 XLSX。

XLSX 增加或保留以下 Sheet：

- `总览`
- `建议操作`
- `商品明细`
- `曝光不足`
- `点击弱`
- `转化弱`
- `高潜力`
- `新品观察`
- `生命周期治理`

### Empty Module Notes

模块为空时输出解释性文案：

- 曝光不足：`暂无达到阈值的曝光不足商品。`
- 点击弱：`暂无达到阈值的高曝光低点击商品。`
- 转化弱：`暂无达到阈值的高访问低转化商品。`
- 高潜力：`暂无达到放量阈值的高潜力商品。`
- 新品观察：`暂无可识别的新进入公域商品，或今日缺少上一日快照。`
- 生命周期治理：`暂无达到长期弱表现阈值的托管商品。`

## Error Handling

- 昨日上下文缺失：记录 run log，继续生成今日基准结论。
- 昨日上下文损坏：记录 run log，继续生成今日基准结论，不影响抓取结果。
- 某个模块无候选商品：输出解释性文案。
- 建议操作为空：输出“暂无需要立即处理的建议操作”。
- 商品 ID 映射缺失：继续回退平台商品 ID。

## Testing

测试覆盖：

- 昨日上下文存在时，结论包含曝光、访问、金额、发货、转化率的较昨日叙述。
- 昨日上下文缺失时，结论包含基准说明且日报继续生成。
- `newProductObservation` 不再固定为空，能从 `new_product` 日差分产生条目。
- `lifecycleGovernance` 能从托管天数和30日弱表现产生条目。
- `recommendedActions` 独立生成，包含 action 和 reason。
- 飞书卡片展示“建议操作”独立板块，并展示 action。
- 飞书卡片展示新品观察和生命周期治理模块。
- Markdown 展示经营结论、建议操作、六个模块。
- XLSX 包含 `建议操作` Sheet。
- 模块为空时输出解释性文案，不只输出“无”。
- `npm test` 通过。
- `npm run build` 通过。

## Success Criteria

- 飞书卡片不再只有漏斗和少量 Top5，而是包含较昨日结论、建议操作和六类模块。
- 推荐操作成为独立板块，老板能直接看到要处理什么。
- 新品观察能列出可识别的新商品。
- Markdown/XLSX 保留完整复盘信息。
- 现有抓取、中文输出路径、飞书发送能力不回退。
